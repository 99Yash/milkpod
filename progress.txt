Issues found and fixed:
1. polar.ts:123,148,171 - Missing timeouts on all three Polar API fetch calls (checkout,
   portal, cancel). Could hang indefinitely if Polar API is unresponsive.
   Fix: Added `signal: AbortSignal.timeout(30_000)` (30s) to all three fetch calls.
2. polar.ts:37-42 - `resolvePlanProductId` always returned first (monthly) product ID,
   ignoring the interval parameter. Yearly checkout would use monthly pricing.
   Fix: Added `interval` parameter; selects ids[0] for monthly, ids[1] for yearly.
   Added validation that the selected ID exists.
3. index.ts:130 - Webhook signature verification uses re-serialized JSON body. Elysia
   auto-parses JSON body, then `JSON.stringify(body)` may produce different bytes than
   the original, causing HMAC verification to fail.
   Fix: Added `onParse` lifecycle hook scoped to `/api/billing/webhook` that returns
   `request.text()` to preserve the raw body string for signature verification.
4. rate-limit.ts:66-88 - `/api/billing` paths were not covered by the rate limiter.
   Checkout, portal, and cancel endpoints had no rate limiting.
   Fix: Added 'billing' rate category (10 req/min burst) and categorize rule for
   `/api/billing` paths.
5. service.ts:7 - Duplicate import: `dailyUsage` imported on separate line from same
   `@milkpod/db/schemas` module. Fix: Merged into single import statement.
6. index.ts:36,74,102 - Provider API errors (from Polar) bubbled up as unhandled
   exceptions, returning 500 with potentially sensitive Polar error text.
   Fix: Wrapped checkout, portal, and cancel provider calls in try/catch blocks.
   Return 502 with safe generic error messages. Log only error message, not full object.
7. index.ts:163 - Webhook error logging used `console.error(..., err)` which could log
   full error object including raw Polar payloads with customer data.
   Fix: Changed to log only `msg` (error message string).

Issues noted (no fix needed):
8. billing.ts:72 - Webhook unique constraint on providerEventId alone (not compound with
   provider). Confirmed from Task 1 review: provider event IDs are unique strings;
   multi-provider collision risk negligible.
9. polar.ts:225-226 - Fallback providerEventId uses `Date.now()` for non-Standard-Webhook

   the webhook-id header. Acceptable as defensive fallback.
10. resolve-provider.ts:5 - Module-level cache for provider singleton uses three-state
    (undefined/null/provider). Clean pattern; avoids re-resolving env on every call.
