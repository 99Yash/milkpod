## Task 13: Standardize API error responses ✅

### Issues found and fixed:
1. **No global error handler existed** — Elysia's built-in validation errors (TypeBox schema failures), unknown route 404s, parse errors, and unhandled exceptions all returned Elysia's default error shapes, which were inconsistent and sometimes leaked internal details. **Fixed:** Created `packages/api/src/middleware/error-handler.ts` with a global `.onError({ as: 'global' })` hook that normalizes all error responses to `{ error: string, code: string }` with correct HTTP status codes (400 for validation/parse, 404 for unknown routes, 500 for unhandled errors).
2. **No typed error classes for services to throw** — Route handlers used `status()` returns for error paths, which is fine but prevents services from signaling errors without coupling to Elysia. **Fixed:** Created `packages/api/src/middleware/errors.ts` with `ApiError` base class and subclasses: `NotFoundError` (404), `ForbiddenError` (403), `UnauthorizedError` (401), `ValidationError` (400), `ConflictError` (409). The global error handler catches these and returns normalized responses.
3. `packages/api/src/modules/billing/index.ts` — **Inconsistent error response shape.** All other modules used `status(code, { message: string, code?: string })`, but billing used `set.status = N; return { error: string, message: string }` — a completely different shape (`error` field instead of `code`, and `set.status` instead of `status()` helper). Frontend code in `pricing-grid.tsx` and `billing-dashboard.tsx` parsed the old `error` field with special string matching. **Fixed:** Converted all 8 billing error returns to use `status()` with `{ message, code }` shape matching the rest of the codebase. Error codes changed from snake_case to SCREAMING_SNAKE_CASE for consistency: `billing_disabled` → `BILLING_DISABLED`, `no_customer` → `NO_CUSTOMER`, `no_subscription` → `NO_SUBSCRIPTION`, `checkout_failed` → `CHECKOUT_FAILED`, `portal_failed` → `PORTAL_FAILED`, `cancel_failed` → `CANCEL_FAILED`, `processing_failed` → `PROCESSING_FAILED`, `invalid_signature` → `INVALID_SIGNATURE`. Removed unused `set` destructuring from all 4 billing handlers.
4. `packages/api/src/modules/billing/index.ts:57,97,133` — **Unsafe error logging in catch blocks.** Three catch blocks used `err instanceof Error ? err.message : err` — the else branch passes the raw error object which could contain connection strings or sensitive data in toString(). **Fixed:** Changed all three to `err instanceof Error ? err.message : String(err)`.
5. `apps/web/src/components/billing/pricing-grid.tsx:107` — Frontend checked `errVal.error === 'billing_disabled'` to detect billing-disabled state. **Fixed:** Updated to check `errVal?.code === 'BILLING_DISABLED'` to match new response shape. Also extracts `errVal?.message` for the toast fallback.
6. `apps/web/src/components/billing/billing-dashboard.tsx:84-93` — Frontend checked `(errVal as { error: string }).error` to match billing error codes. **Fixed:** Updated to use `errVal?.code` matching with new SCREAMING_SNAKE_CASE codes. Also extracts `errVal?.message` for the toast fallback.

The new global error handler returns `{ error: string, code: string }` (using `error` not `message`). **Fixed:** Added fallback check for `.error` field after `.message` check, so both shapes are handled.

### Issues noted (no fix needed):
1. **`status()` returns bypass the onError hook** — Existing route handlers return `status(404, { message: '...' })` which Elysia treats as typed responses, not errors. These don't flow through `.onError()`. Task 37 will refactor these to throw `NotFoundError`/`ForbiddenError` via a `resolveOwned()` helper, which the error handler will then catch.
2. **Chat module model gating uses `new Response()`** — `chat/index.ts:63-70` constructs a raw `Response` for the 402 MODEL_NOT_ALLOWED error. ## Task 14: Collapse N+1 queries into JOINs ✅

### Issues found and fixed:
1. `packages/api/src/modules/assets/service.ts:191-223` — `AssetService.getWithTranscript()` made 3 sequential queries: (1) `getById` → SELECT from media_assets, (2) SELECT all transcripts for the asset, (3) SELECT all segments for all transcripts. These 3 round trips are unnecessary since the data can be fetched in a single query. **Fixed:** Replaced with a single SELECT using LEFT JOIN: `media_assets LEFT JOIN transcripts LEFT JOIN transcript_segments`, ordered by `desc(transcripts.createdAt), desc(transcripts.id), transcriptSegments.startTime`. Rows are grouped in JS by transcript ID (Map preserves insertion order = createdAt DESC), then the "prefer transcript with segments" logic iterates the map to find the first transcript with segments — identical behavior to the original 3-query version.

2. `packages/api/src/modules/threads/service.ts:83-94` — `ThreadService.getWithMessages()` made 2 sequential queries: (1) `getById` → SELECT from qa_threads, (2) SELECT all messages for the thread. **Fixed:** Replaced with a single SELECT using LEFT JOIN: `qa_threads LEFT JOIN qa_messages`, ordered by `qaMessages.createdAt`. Messages are extracted from joined rows via `flatMap` with null filtering.

3. `packages/api/src/modules/shares/service.ts:103-131` — `ShareService.getSharedResource()` asset path made 3 sequential queries after token validation: (1) SELECT asset, (2) SELECT transcript, (3) SELECT segments. **Fixed:** Replaced with a single SELECT using LEFT JOIN: `media_assets LEFT JOIN transcripts LEFT JOIN transcript_segments`, ordered by `desc(transcripts.createdAt), desc(transcripts.id), transcriptSegments.startTime`. Uses the latest transcript and filters segments to only those belonging to it. Also fixed non-deterministic transcript selection — old code had no ORDER BY on the transcript query, new code consistently selects the latest.

4. `packages/api/src/modules/shares/service.ts:134-166` — `ShareService.getSharedResource()` collection path made 2 sequential queries after token validation: (1) SELECT collection, (2) SELECT items INNER JOIN assets. **Fixed:** Replaced with a single SELECT using LEFT JOIN: `collections LEFT JOIN collection_items LEFT JOIN media_assets`, ordered by `collectionItems.position`.
