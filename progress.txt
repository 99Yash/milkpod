## Task 14: Collapse N+1 queries into JOINs ‚úÖ

### Issues found and fixed:
1. `packages/api/src/modules/assets/service.ts:191-223` ‚Äî `AssetService.getWithTranscript()` made 3 sequential queries: (1) `getById` ‚Üí SELECT from media_assets, (2) SELECT all transcripts for the asset, (3) SELECT all segments for all transcripts. Rows are grouped in JS by transcript ID (Map preserves insertion order = createdAt DESC), then the "prefer transcript with segments" logic iterates the map to find the first transcript with segments ‚Äî identical behavior to the original 3-query version.

Messages are extracted from joined rows via `flatMap` with null filtering.

3. `packages/api/src/modules/shares/service.ts:103-131` ‚Äî `ShareService.getSharedResource()` asset path made 3 sequential queries after token validation: (1) SELECT asset, (2) SELECT transcript, (3) SELECT segments. **Fixed:** Replaced with a single SELECT using LEFT JOIN: `media_assets LEFT JOIN transcripts LEFT JOIN transcript_segments`, ordered by `desc(transcripts.createdAt), desc(transcripts.id), transcriptSegments.startTime`. Uses the latest transcript and filters segments to only those belonging to it. Also fixed non-deterministic transcript selection ‚Äî old code had no ORDER BY on the transcript query, new code consistently selects the latest.

4. `packages/api/src/modules/shares/service.ts:134-166` ‚Äî `ShareService.getSharedResource()` collection path made 2 sequential queries after token validation: (1) SELECT collection, (2) SELECT items INNER JOIN assets. **Fixed:** Replaced with a single SELECT using LEFT JOIN: `collections LEFT JOIN collection_items LEFT JOIN media_assets`, ordered by `collectionItems.position`. ## Task 15: Wrap multi-step writes in transactions ‚úÖ

### Issues found and fixed:
1. `packages/api/src/modules/podcasts/service.ts:20-47` ‚Äî `addFeed()` performed feed insert + episode upsert as two separate operations. If the episode insert failed after the feed was created, the database would contain a feed row with no episodes, and there was no mechanism to detect or clean this up. **Fixed:** Wrapped both operations in `db().transaction()`. The `parseFeed()` network call is kept outside the transaction to avoid holding a DB connection open during the HTTP request. Modified `upsertEpisodes()` to accept a `tx` parameter (typed via `DbTx` extracted from the transaction callback signature).
2. `packages/api/src/modules/podcasts/service.ts:133-159` ‚Äî `refreshFeed()` performed feed metadata update + episode upsert as two separate operations. If the episode upsert failed, the feed metadata would already be updated but episode counts could be stale. **Fixed:** Wrapped feed update + episode upsert in `db().transaction()`. The `parseFeed()` and `getFeed()` calls remain outside the transaction (read + network).
3. `packages/api/src/modules/chat/index.ts:89-118` ‚Äî After reserving words from the daily quota, if `ThreadService.create()` returned null or threw an error, the reserved words were permanently leaked. The `onFinish` callback on `createChatStream` handles releasing unused words during normal operation, but if the stream never starts, the callback never fires. **Fixed:** Added `releaseReserved()` helper that releases reserved words (non-admin only). Called on: (a) `ThreadService.create` returns null, (b) `ThreadService.create` throws, (c) `threadId` still null after try block (defensive). The helper's `.catch(() => {})` suppresses release failures since we're already in an error path.
4. `packages/api/src/modules/podcasts/episode-pipeline.ts:39-61` ‚Äî `AssetService.create()` and `PodcastService.linkAsset()` were two separate operations. If `linkAsset()` failed after asset creation, the episode retained `assetId: null` while an orphaned asset existed. On retry, the pipeline would check `!assetId`, find null, and create yet another orphaned asset. **Fixed:** Wrapped asset insert + episode link in `db().transaction()`. Inlined the asset insert (via `tx.insert(mediaAssets)`) rather than modifying `AssetService.create` to accept a tx parameter, since the pipeline only needs the asset ID. Used `.returning({ id: mediaAssets.id })` to select only the needed column.
5. `packages/api/src/modules/podcasts/episode-pipeline.ts:6` ‚Äî `AssetService` import became dead code after inlining the insert in the transaction. **Fixed:** Removed the unused import.

### Issues noted (no fix needed):
1. **Chat thread auto-creation is not a single DB transaction with word reservation** ‚Äî The plan specified "wrap thread create + word reservation in transaction." However, `UsageService.reserveWords` uses `pg_advisory_xact_lock` inside its own `db().transaction()`. Wrapping both in a single outer transaction would create nested transactions (savepoints) and complicate the advisory lock semantics. Instead, the pragmatic fix releases reserved words on any failure between reservation and stream start. This achieves the same atomicity guarantee (no leaked quota) without changing the advisory lock pattern.
2. **`PodcastService.linkAsset()` method is still public** ‚Äî It's now partially redundant for the episode pipeline (which uses the transaction directly), but it may be needed for other callers or future use. Left as-is.
3. **`upsertEpisodes` still uses select-filter-insert instead of INSERT ON CONFLICT** ‚Äî Task 29 specifically addresses converting this to `INSERT ... ON CONFLICT (guid) DO NOTHING`. Not in scope for Task 15.

### Library verification:
- **Drizzle transaction API**: Verified via `pg-core/db.d.ts` that `db().transaction()` callback receives `PgTransaction<TQueryResult, TFullSchema, TSchema>` which extends `PgDatabase` ‚Äî has the same `select`/`insert`/`update`/`delete`/`execute` API as the main `db()` instance.
