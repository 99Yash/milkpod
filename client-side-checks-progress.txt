## 1.1 Daily Word Gate in ChatPanel

**Files changed:**
- `apps/web/src/components/chat/chat-panel.tsx`
  - Added `isWordBudgetExhausted` derived from `wordsRemaining === 0`
  - Added word budget check in `handleSubmit` (after model block check, before send)
  - Added same check in suggestion button click handlers
  - Toast: "Daily word limit reached. Resets at midnight UTC." with "View plans" action (8s duration)
  - Added `isWordBudgetExhausted` to `handleSubmit` dependency array

**Decisions:**

- Used direct `toast.error()` pattern matching the existing `isModelBlocked` check, rather than `handleUpgradeError`, since the server returns 429 (not 402) for word budget exhaustion.

## 1.3 Comments Quota Pre-Check in CommentsTab

**Files changed:**
- `apps/web/src/components/comments/comments-tab.tsx`
  - Added imports: `checkQuotaLocal`, `incrementMonthlyUsage` from `~/lib/plan-cache`
  - Added `checkQuotaLocal('comments')` pre-check at the top of `handleGenerate`, before `setGenerating(true)` — if `!allowed`, calls `handleUpgradeError` with `QUOTA_EXCEEDED` code and returns early
  - On success, extracted `generated` variable from `(data as Comment[]) ?? []`, then calls `incrementMonthlyUsage('comments', generated.length)` to optimistically update the local counter

**Decisions:**
- Admin bypass: `checkQuotaLocal` returns `null` when the cache hasn't been populated, and admin users on `pro`/`team` plans have high enough quota limits that the check passes naturally. No explicit admin check needed.
- When `checkQuotaLocal` returns `null` (plan or usage not yet loaded), the request proceeds to the server — server remains the authority (per PRD principle 1).
- Pre-check is placed before `setGenerating(true)` so no loading spinner flashes when the user is over quota.
- Followed the exact same pattern as `url-input-form.tsx` (UrlInputForm): `checkQuotaLocal` → early return with `handleUpgradeError` → `incrementMonthlyUsage` on success.
- Only increments the counter when `generated.length > 0` to avoid no-op increments.

**Decisions:**
- Chose Option A (extract count from collections list query) over Option B (billing summary) — simpler, no backend change needed, and the library page already fetches the full list on mount.
